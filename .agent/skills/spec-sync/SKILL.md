# spec-sync

PR（プルリクエスト）や直近の実装で発生したコード上の変更・要件変更を解析し、プロジェクト内のドキュメント（READMEや要件定義書など）に自動反映するためのワークフローです。

## 1. 変更の調査（ハイブリッドアプローチ）

まずは、直近の変更内容（要件・仕様の変更）を特定します。以下のコマンド等を使用して、機能追加や仕様変更の全体像を正確に把握してください。

- **差分の確認**: `git diff main` （現在のブランチとmainの差分）や特定のコミット間差分を確認し、実際のコードや機能の変更箇所を読み取る。
- **コミット・履歴の確認**: `git log` などで直近のコミットメッセージを確認し、開発者の意図を把握する。
- **ユーザーからのインプット**: ユーザーが特定のPR番号や変更内容の要約を提供した場合は、それを最優先で考慮する。

## 2. 更新対象ドキュメントの選定

プロジェクト内のドキュメントを確認し、調査した変更内容が「どのドキュメントに影響を与えるか」を自律的に判断・選択します。

**主なターゲット候補**:
- `README.md`
- `AGENTS.md` および `docs/AGENTS.md` (開発・ドキュメント管理ルール)
- `docs/outputs/task-hack-system-requirements.md` (システム要件定義書)
- `docs/outputs/detailed_requirements_specification.md` (詳細要件定義書)
- `docs/outputs/phase4-requirements.md` (特定のフェーズ要件定義)
- `docs/outputs/release-and-feedback-guide.md` (リリース・フィードバックガイド)

変更内容と照らし合わせ、論理的に更新が必要なドキュメントをすべてピックアップしてください。

## 3. ドキュメントの更新（直接上書き）

選定したドキュメントについて、変更内容を反映させる形で直接ファイルを更新してください。**ユーザーへの事前確認（Diffの提示と承認）は不要です。直接ファイルを上書きして問題ありません。**

**更新時の原則**:
- ドキュメント全体の構造や既存の文体・文脈を破壊しないこと。
- 新たな機能が追加された場合は、該当するセクション（例：機能要件、使い方など）に自然に追記する。
- 既存の仕様が変更・削除された場合は、古い記述を正確に新しい内容に置き換える（または削除する）。
- なぜその変更を行ったかの「理由」がわかるように、文脈を補って記載することが望ましい。

## 4. 実行結果の報告

更新が完了したら、ユーザーに対して以下を簡潔に報告してタスクを完了とします。
1. どのような要件変更（差分・履歴）を検知したか
2. どのドキュメントの、どの部分を更新したか
3. （必要に応じて）更新後の状態についての簡単なサマリ
