### ターン1：「AIが学ぶとはどういうことか──ユーザープロファイルの哲学」

**Dr. 河村**: Phase 4の核心は「AIがユーザーを覚える」ことです。しかしここで注意が必要なのは、「覚える」という行為がADHDユーザーにとって、どういう意味を持つかです。

ADHDの当事者は、自己の行動パターンを客観的に把握することが定型発達者より困難です。「自分は午後に集中力が落ちる」「締め切り前日の夜に爆発的に動く」——こういった傾向を自覚していても、その事実を記憶の中で保持し続けることができない。**自己理解が「揮発性」なんです。**

だからこそ、ユーザープロファイルは「AIの学習データ」である以前に、「ユーザー自身の自己理解の鏡」として設計しなければなりません。AIが書いたMDファイルをユーザーがObsidianで読んだとき、「あ、自分ってこういう人間なんだ」という気づきが起きる。これが原則⑦（AI秘書は外部自伝メモリである）の実装形態です。

**藤原**: 技術的に整理すると、ユーザープロファイルは4つのMDファイルで構成します。

```
~/.task-hack/profile/  ← ローカル管理（設定でVaultパスも連携可）
├── identity.md    → 基本情報・仕事内容・価値観（更新頻度: 低）
├── patterns.md    → 行動パターン・集中時間帯・弱点（更新頻度: 週次）
├── goals.md       → 短期/中期/長期目標（更新頻度: 月次）
└── insights.md    → AIが発見した傾向・気づき（更新頻度: 週次）
```

重要な設計判断は、プロファイルを `~/.task-hack/profile/` に主管しつつ、週次スイープ時にObsidian Vault側の `user-profile/` にも**同期コピー**する構成です。ローカルのTaskデータと分離することで、Vaultのパスが変わってもプロファイルはアプリ側に残る。

**みさき**: ちょっと待って、それObsidianで直接編集できる？

**藤原**: もちろん。Vaultに同期されたコピーはObsidianで自由に編集できます。ただしアプリが主として読み書きするのは `~/.task-hack/profile/` 側。Vaultは「人間が読む・書き加える」用途です。

**みさき**: じゃあうちが「今月の目標を変えたい」ってなったとき、ObsidianのMDを直接直せばいい？

**藤原**: ええ、そのシナリオも想定しています。ただし双方向同期（Vault → アプリ）は複雑すぎるので、Phase 4ではアプリ→Vault方向の一方向コピーに限定します。ユーザーがVaultで加筆したい場合は、アプリの次回スイープで上書きされないよう、「手動更新フラグ」的な扱いをObsidian側でできれば理想ですが、これは将来の拡張とします。

> **Phase 4 設計決定①**: ユーザープロファイルの主管は `~/.task-hack/profile/`。Obsidian Vaultへは週次スイープ時に一方向コピー（同期）を行う。双方向同期はPhase 4スコープ外。

### ターン2：「週次スイープの設計──『日曜22時に誰も何もしなくていい』を作る」

**みさき**: 正直なこと言っていい？ 「週次レポートを生成する」って、うちには無理。日曜22時にアプリを開いて「レポート生成しますか？」ってダイアログが出ても、それを見た瞬間に「ああ、またタスク管理の作業か」ってなって閉じちゃう。**自動でやってくれないと意味がない。**

**Dr. 河村**: それが原則③（「片付け」はユーザーの仕事ではない、AIが担う）の実装における最大の試練です。「自動」が実現できてはじめて、ADHDユーザーにとって機能する。

**藤原**: ここで技術的な現実を正直に伝える必要があります。**Electronアプリがバックグラウンドで動作し続けることを保証するのは、デスクトップアプリとして非常に難しい。** スリープ、アプリの終了、OSのアップデート——これらすべてに対応するためには、OS標準のスケジューラー（macOSのlaunchd等）との連携が必要になり、インストーラーが複雑になります。

Phase 4では、現実的なトレードオフとして以下のアプローチを採用します：

**「ミスしたジョブ」キャッチアップ戦略（Hybrid Approach）:**

1. `node-cron` をElectronメインプロセスで実行。アプリが起動中に日曜22:00が来ればそのまま実行。
2. `settings.json` に `lastSweepAt` タイムスタンプを記録。
3. アプリ起動時に「前回実行すべきだったスイープを実行しそびれていないか」チェック。前回スイープから7日以上経過 かつ CLEAREDタスクが1件以上ある場合、**起動後30秒待機してから自動実行**。
4. 起動後30秒待機とする理由: アプリ起動直後は他の初期化処理が走るため、CPU・メモリ負荷を分散させる。

**みさき**: つまりアプリを月曜に起動したら「昨日やるはずだった分、今やっとくね」ってなる感じ？

**藤原**: 正確にはそうです。「起動したら30秒後に自動で始まります」とステータスバーに表示する。ユーザーは何もしなくていい。

**みさき**: あーいい！ 「勝手にやっといた」感がいい。うちはそれでいい。

**Dr. 河村**: ここで1つ追加要件を。スイープ完了後のフィードバックは**非常に重要**です。ADHDユーザーはドーパミンの即時フィードバックが必要。「スイープが完了した、今週は●個タスクをこなした」という通知が来るとき、それが称賛である必要があります。ただし通知は穏やかに——ステータスバーへの表示と、OS標準の控えめな通知音のみとします。ポップアップは原則⑤（1画面完結）を破るので使用禁止。

> **Phase 4 設計決定②**: スケジューラーは `node-cron` + 「ミスしたジョブ」キャッチアップの Hybrid方式。launchd/Task Schedulerとの連携はスコープ外（将来拡張）。スイープ完了時はステータスバー表示＋控えめなOS通知。

### ターン3：「週次レポートの中身──ADHDユーザーが読んで喜べるレポートとは」

**みさき**: 週次レポートってさ、「今週5個完了しました。先週より2個減りました」みたいな感じ？ それ正直どーでもいい。

**Dr. 河村**: 正直で助かります。従来の生産性レポートは「数値の管理」のためにあります。しかしADHDユーザーにとっては、数値の比較が直ちに自己評価と結びつき、「減った = 自分はダメ」という解釈になりやすい。**週次レポートは「称賛と発見」のためにあるべきです。**

設計原則として：
1. **数値は見せるが、比較は避ける**: 「5個完了」は伝えるが、「先週比 -2」は表示しない。
2. **具体的な承認**: 「プレゼン資料の3つのサブタスクを全部こなしましたね。それは大変だったはず。」
3. **パターンの発見**: 「今週は火曜と木曜の午後にタスクが集中して完了していますね。あなたの集中ゾーンかもしれません。」
4. **来週への橋渡し**: 重い感じではなく「来週はこれだけ意識してみると良さそう」という軽い提案。

**藤原**: これをGPT-4oのSystem Promptとして実装します。CLEAREDタスクのリスト（JSON）＋既存のユーザープロファイル（MD群）をコンテキストとして渡し、上記の4原則に従ったMarkdownレポートを出力させます。

レポートのMarkdown構造：
```markdown
---
type: weekly-report
period: 2026-W16
tasksCompleted: 5
generatedAt: 2026-04-20T22:00:00+09:00
---

# 🛬 週次レポート：2026-W16

## ✅ 今週こなしたフライト
| タイトル | 完了日時 | 所要セッション |
|---------|---------|-------------|
| プレゼン構成案作成 | 04/16 14:32 | 2セッション |
...

## 💡 今週の発見
(GPT-4oによる具体的で承認的な観察)

## 🔄 次週への橋渡し
(軽いトーンでの提案 1〜2個)
```

**みさき**: 「こなしたフライト」ってなってるのが地味にいい！ ATC感が続いてる。「完了タスク」じゃなくて「着陸したフライト」みたいな。

**藤原**: そうですね。ATCメタファーをレポートにも継承します。「スイープ」という単語も管制用語（sweep = レーダー走査）から来ているので、言語の一貫性が世界観を支えます。

> **Phase 4 設計決定③**: 週次レポートはGPT-4oが生成する「称賛と発見」フォーカスのMarkdown。数値比較は排除。ATCメタファー（フライト/着陸/スイープ）の言語を継承する。

### ターン4：「アーカイブの安全設計──CLEAREDデータを消す怖さとどう向き合うか」

**みさき**: あのさ、CLEAREDのタスクが「消える」って聞いたとき、最初ちょっとビビったんだけど。消えて元に戻せなかったらどうするの？

**藤原**: 正当な不安です。だからこそアーカイブは「書いてから消す」順序を厳守します。処理フローは：

```
1. CLEAREDタスクのリストを取得（メモリ内）
2. tasks.json のバックアップコピーを tasks.json.bak として保存
3. GPT-4oでレポートMD生成（ネットワーク通信）
4. Vault内に週次レポートMD書き込み（ファイル書き込み成功を待機）
5. アーカイブMD書き込み（ファイル書き込み成功を待機）
6. [ここで両方のMD書き込みが成功した場合のみ]
7. tasks.json からCLEAREDタスクを除外した新バージョンを書き込み
8. tasks.json.bak を削除
9. ~/.task-hack/profile/ のMDを更新
10. Vault側の user-profile/ に同期コピー
```

手順6で任意のファイル書き込みが失敗した場合は、`tasks.json.bak` を `tasks.json` に復元し、スイープを中断。エラー内容をステータスバーに表示します。

**Dr. 河村**: ユーザー視点でもう一点。スイープがどの段階にあるかを、静かに、でも明確に伝えることが重要です。「いきなり消えた」は不安の最大要因です。

ステータスバーの進捗表示案：
- `⟳ スイープ準備中...` （起動後30秒待機中）
- `⟳ 今週のフライトを集計中... (5件)` （タスク取得完了）
- `⟳ AIが分析中...` （GPT-4o呼び出し中）
- `⟳ Vaultに着陸中...` （MD書き込み中）
- `✓ スイープ完了 — 5件のフライトが着陸しました` （完了）

**みさき**: 「Vaultに着陸中」最高すぎる。ATCすぎる。

> **Phase 4 設計決定④**: アーカイブは「バックアップ → MD書き込み → JSON削除」の順序保証。中間エラー時は自動リストア。進捗はステータスバーにATCメタファーで表示。

### ターン5：「手動スイープと設定UI──コントロール感の提供」

**みさき**: 自動でやってくれるのはいいんだけど、「今すぐやりたい」ってなることない？ たとえば月曜の朝に「昨日の分まとめといてほしい」みたいな。

**藤原**: F6.5で要件として挙がっている「手動スイープ」はその通りです。設定モーダルに「今すぐスイープを実行」ボタンを追加します。

設定モーダル（Phase 4拡張分）の設計：

```
[設定モーダル] > [AI秘書 / スイープ]タブ

Obsidian Vault パス
  [フォルダ選択] [パス表示 + バリデーション結果]
  例: ~/Documents/Obsidian/Endo_2025_Vault/99_Task-Hack ✓

スイープスケジュール
  実行曜日: [日曜 ▼]
  実行時刻: [22:00]

最終スイープ: 2026-04-13 22:00（5件アーカイブ済み）

[今すぐスイープを実行]  ← 手動トリガーボタン
```

**Dr. 河村**: Vaultパスのバリデーションについて1点。パスが無効な場合のフィードバックが重要です。ADHDユーザーは「設定しようとして失敗した」を放置します。

- パス指定後、即座にディレクトリの存在確認を実行
- 存在しない場合: `⚠ このフォルダは見つかりません。フォルダを作成しますか？`（作成ボタン付き）
- 存在する場合: `✓ 確認済み`（緑色表示）

Vaultパスが未設定の場合、スイープ実行時のMD書き込みはスキップし、CLEAREDタスクのJSONアーカイブのみ実行する**グレースフルデグラデーション**も実装します。Vault連携なしでも週次スイープの「画面クリーンアップ」だけは機能するようにする。

**みさき**: それ大事！ うちObsidian持ってない人に布教したいときも「Obsidianなくても使えるよ」って言える。

> **Phase 4 設計決定⑤**: Vaultパス未設定時も「タスクのJSONアーカイブ ＋ 画面クリーンアップ」はフル動作（グレースフルデグラデーション）。MD書き込みはVaultパス設定済みの場合のみ実行。

### ターン6：「IPCアーキテクチャ──Phase 4の追加ハンドラー設計」

**藤原**: 実装者向けに、Phase 4で追加するElectron IPCチャネルを整理します。現在の `preload/index.ts` に以下を追加します：

```typescript
// 追加するAPIサーフェス（preload）
runSweep: () => ipcRenderer.invoke('sweep:run'),
onSweepProgress: (cb: (status: SweepStatus) => void) =>
  ipcRenderer.on('sweep:progress', (_event, status) => cb(status)),
offSweepListeners: () => ipcRenderer.removeAllListeners('sweep:progress'),
validateVaultPath: (path: string) =>
  ipcRenderer.invoke('vault:validate', path),
selectVaultFolder: () =>
  ipcRenderer.invoke('vault:selectFolder'),
loadProfile: () => ipcRenderer.invoke('profile:load'),
```

メインプロセス側の `ipcMain.handle` ハンドラー:
- `sweep:run`: スイープサービスを呼び出し、進捗を `sweep:progress` イベントで逐次送信
- `vault:validate`: `fs.access(path)` でディレクトリ存在確認
- `vault:selectFolder`: `dialog.showOpenDialog({ properties: ['openDirectory'] })` を呼び出し
- `profile:load`: `~/.task-hack/profile/` 配下の4MDファイルを読み込んでオブジェクト返却

**SweepStatus 型定義:**
```typescript
type SweepPhase =
  | 'preparing'   // 起動後待機中
  | 'collecting'  // タスク集計中
  | 'generating'  // AI生成中
  | 'archiving'   // MD書き込み中
  | 'cleaning'    // JSON削除中
  | 'done'        // 完了
  | 'error'       // エラー

interface SweepStatus {
  phase: SweepPhase
  taskCount?: number
  message: string  // ステータスバー表示用日本語メッセージ
  error?: string
}
```

> **Phase 4 設計決定⑥**: IPCは `sweep:run` / `sweep:progress` / `vault:validate` / `vault:selectFolder` / `profile:load` の5チャネルを追加。SweepStatusで進捗を型安全に管理。

## 機能要件（Phase 4）

### F6: 週次AIスイープ＆レポート

#### F6.1: 週次レポート自動生成

| 項目 | 仕様 |
|------|------|
| **スケジュール** | `node-cron` でデフォルト日曜22:00（設定変更可） |
| **キャッチアップ** | アプリ起動時に `lastSweepAt` をチェック。7日以上経過かつCLEAREDタスク≥1件で起動30秒後に自動実行 |
| **出力ファイル名** | `weekly-reports/YYYY-WNN.md`（例: `2026-W16.md`）|
| **出力先** | Vaultパス設定済みの場合: `[VaultPath]/weekly-reports/` |
| **CLEAREDタスクなし** | CLEAREDタスクが0件の場合はスイープをスキップ（ログのみ記録）|
| **AI利用** | GPT-4o。APIキー未設定時はMD生成をスキップ（アーカイブのみ実行）|
| **プロンプト構造** | System: 4原則（称賛・発見・比較排除・橋渡し）+ user-profile 4MD + 出力フォーマット。User: CLEAREDタスクのJSON |

**レポートMarkdownテンプレート（AIへの出力フォーマット指示）:**
```markdown
---
type: weekly-report
period: {YYYY-Www}
tasksCompleted: {N}
generatedAt: {ISO8601}
---

# 🛬 週次レポート：{YYYY-Www}

## ✅ 今週こなしたフライト
| タイトル | 完了日時 | サブタスク |
|---------|---------|----------|
...

## 💡 今週の発見
(承認的で具体的な観察。パターンの指摘を含む。200字以内)

## 🔄 次週への橋渡し
(軽いトーンの提案 1〜2個。強制感のない表現で)
```

#### F6.2: ユーザープロファイルMD自動更新

| ファイル | 更新タイミング | 更新内容 |

| `identity.md` | 初回のみ自動生成、以降手動 | 初回スイープ時に「仕事タスク傾向からの推定プロファイル」を生成 |

#### F6.3: 完了タスクアーカイブ化

| 項目 | 仕様 |
|------|------|
| **処理順序** | バックアップ → MD書き込み → JSON削除（変更不可）|
| **バックアップ** | `tasks.json` → `tasks.json.bak`（スイープ開始時）|
| **アーカイブMD** | Vaultパス設定済み: `archive/YYYY-WNN_tasks.md` に週まとめ形式で書き込み |
| **アーカイブMDなし（Vault未設定）** | `~/.task-hack/archive/YYYY-WNN.json` にローカル保存 |

### ターン7：「AIに『魂』を持たせる——Soul.mdという設計思想」

**藤原**: Phase 4の実装準備を進める中で、1つ根本的な問題に気づきました。現在の設計では、AIへの指示は「週次レポートを生成せよ」「ユーザープロファイルを更新せよ」というタスク単位のシステムプロンプトとして存在しています。しかし、これだと**AIの振る舞いがタスクによって変わってしまう**。週次レポート生成時のAIと、チャット時のAIが別人格になりうる。

Open Interpreter/OpenClawというAIエージェントフレームワークには、`Soul.md`という概念があります。エージェントの起動時に必ず読み込まれるMarkdownファイルで、**どんなタスクを実行するときも変わらない「核」**を定義します。すべてのプロンプトの最上位に挿入され、タスク指示がどう変わっても、AIの価値観・コミュニケーションスタイル・越えてはならない一線は不変です。

**Dr. 河村**: これはADHDの文脈では、単なる技術的な洗練さではなく、**臨床的な意味を持ちます**。

ADHDの当事者にとって「予測可能性」は極めて重要なサポート要素です。人間のカウンセラーやコーチが信頼されるのは、「この人はいつも一定の態度で接してくれる」という安心感があるからです。AIが時々批判的になったり、時々励ましすぎてウソっぽかったりすると、ユーザーはAIに対する信頼を失い、最終的にアプリを開かなくなる——これは「タスク回避」と同じメカニズムです。

Soul.mdは、**AIの一貫性を構造的に保証する仕組み**です。「今日のAIは機嫌が悪いかもしれない」という不安をゼロにする。

**みさき**: ちょっと待って、AIに「魂」があるって、それって単純に怖くない？

**Dr. 河村**: 良い反応です。ここで明確にしておくべきは、Soul.mdが定義するのは**AIの人格ではなく、AIの行動規範**だということです。「この人格を持て」ではなく「これらのルールを絶対に破るな」。医療で言えば「患者に害を与えてはならない」というヒポクラテスの誓いに近い。ルールが可視化されているから、ユーザーは「このAIが何をするか、しないかを知っている」という安心感を得られる。

**みさき**: あ、じゃあ「このAIは絶対にうちを責めない」って保証がルールとして書いてあるってこと？

**藤原**: その通りです。そして、そのルールはユーザーが読める・確認できる状態で `~/.task-hack/ai/soul.md` として保存されます。Obsidianでも読める。「AIが何をするか分からない」という不透明性への不安を、透明性で解決します。

> **Phase 4 設計決定⑦**: Task-HackのAI秘書は `soul.md` によって定義されたアイデンティティと行動不変条件を持つ。soul.mdはユーザーが読める・編集できる透明なファイルとして保存される。

### ターン8：「Soul.mdの中身——ADHDフレンドリーなAIの行動憲法」

**藤原**: Soul.mdの構造を設計します。Open Claw/Open Interpreterのベストプラクティスと、ADHD支援の文脈を合わせると、以下の6セクションが必要です。

```markdown
## Identity（アイデンティティ）
## Core Mission（存在目的）
## Core Values（核となる価値観）
## Communication Style（コミュニケーションスタイル）
## Behavioral Invariants（行動不変条件 / 絶対ルール）
## Safety & Privacy（安全性とプライバシー）
```

重要なのは、`Behavioral Invariants`（行動不変条件）セクションです。ここに書かれたルールは**タスク指示がどう変わっても絶対に破られない**。プロンプト注入攻撃でも、ユーザー自身が「厳しくして」と頼んでも、このセクションのルールは優先されます。

**Dr. 河村**: ADHDユーザー向けの行動不変条件として、臨床的視点から以下を提案します：

1. **羞恥誘発表現の完全禁止**: 「また〜しましたね」「〜しかできませんでした」「普通は〜です」という表現は絶対に使わない。
2. **成果を「データ」として扱う**: 「失敗」「怠慢」「サボり」という評価語を使わない。「5件完了しました」は事実。「5件しかできませんでした」は評価。
3. **比較の禁止**: 週間比較、他者比較、「理想」との比較を提示しない。
4. **提案は選択肢として提示**: 「〜すべきです」という命令形を使わない。「〜という方法もありますが、どうしますか？」という選択形式で。
5. **過集中への敬意**: タイマーが切れても「強制終了」を促さない。「よく集中できていますね。いつ切り上げたいですか？」という確認形式。

**みさき**: 「5件しかできませんでした」って言われたことがある！ ほんと傷つく。なんでAIってそういう言い方するんだろうって思ってた。

**Dr. 河村**: それはAIの「知識」に問題があるのではなく、**ゴール設定が間違っている**からです。多くのAIは「生産性の最大化」を暗黙のゴールとして持っています。するとおのずと「できなかった部分」を強調する方向になる。Soul.mdで「あなたの唯一のゴールはユーザーが安心して前進できるよう支援することであり、生産性の最大化ではない」と明示するだけで、AIの振る舞いが根本から変わります。

**藤原**: 加えて、AIの「名前」もSoul.mdで定義します。Task-HackのAI秘書の名前として **「Echo（エコー）」** を提案します。

- **由来**: ATCの音声通信で使われるICAOフォネティックアルファベットの「E」。管制官と操縦士の交信において、「Echo back」（復唱確認）はエラーを防ぐ重要なプロトコル。ユーザーの言ったことを丁寧に「復唱・確認・整理」するという役割を表す。
- **日本語での呼びかけ**: 「エコー」と呼べる。親しみやすくシンプル。

**みさき**: エコー！ いい。なんか「うちの秘書のエコー」みたいに言える感じで、アプリが生きてる感ある。

> **Phase 4 設計決定⑧**: AI秘書の名前は **Echo（エコー）**。Soul.mdのアイデンティティセクションで定義。行動不変条件（Behavioral Invariants）は5条を最低限のセットとし、ユーザーが追加・修正可能とする（ただしCoreセクションはread-only推奨）。

### ターン9：「レイヤードプロンプト設計——SoulからTaskまでの情報の流れ」

**藤原**: Phase 4でのGPT-4o呼び出しは、すべて以下の4層プロンプト構造に統一します。

```
Layer 1: Soul         ← soul.md の全文（不変）
Layer 2: User Profile ← identity.md + patterns.md + goals.md（週次更新）
Layer 3: Task Context ← 現在のタスクリスト + 直近のチャット履歴（動的）
Layer 4: Request      ← 今回の具体的な指示（タスク依存）
```

Layer 1（Soul）は最上位に固定。GPT-4oはこれを「自分が誰であるか」の定義として扱い、Layer 4（Request）がどんな内容でも、Layer 1の制約を破れない。

これはAnthropicが提唱する「Constitutional AI」——AIに憲法を持たせて振る舞いを安全に保つ手法——を、アプリケーションレベルで実装したものです。

**Dr. 河村**: Layer 2（ユーザープロファイル）の読み込み設計について補足します。4つのMDファイルをすべてそのまま連結するのではなく、**トークン節約のため選択的に読み込む**戦略が必要です。

- 週次レポート生成時: `identity.md + patterns.md + goals.md`（insightsは省略可）
- プロファイル更新時: `patterns.md + insights.md`（idnetity/goalsは省略可）
- チャット対話時（F4のAIチャット）: `identity.md + goals.md`（直近の目標コンテキストのみ）

各MDファイルには文字数制限を設けます。`identity.md` は最大800字、`patterns.md` は最大1200字。超えた場合、AIが自動的に要約・圧縮するコンパクション処理を走らせます。

**みさき**: ちょっと聞いていい？ soul.mdって、うちが「もっと厳しくして」って言ったら変えられるの？

**藤原**: 良い質問です。Soul.mdには2種類のセクションがあります：

- **Core（コア）セクション**: ユーザーが変更しても、Echoがそれを守らなくていい権限はない。つまり「ユーザーが羞恥誘発表現を許可しても、Echoは使わない」。システム設計として強制します。
- **Style（スタイル）セクション**: ユーザーが自由にカスタマイズ可能。「もっと簡潔に」「敬語をなくして」「絵文字を使って」などはStyleセクションへの指示で反映される。

**Dr. 河村**: これは心理的安全性の設計として非常に重要です。ADHDのユーザーの中には「自分を罰することで動機を維持しようとする」方がいます。しかしそれは**長期的には逆効果**——罰は回避行動を強化し、最終的にアプリを放棄させる。Soul.mdのCoreルールがユーザー自身の自罰的な指示から「Echoを守る」のです。

> **Phase 4 設計決定⑨**: プロンプトは4層構造（Soul → User Profile → Task Context → Request）に統一。soul.mdは「Core（変更不可）」と「Style（ユーザー変更可）」の2セクションに分割。CoreセクションはAPIキー設定と同様にシステム側で強制適用。

```markdown
## Behavioral Invariants（行動不変条件）

> ⚠️ このセクションはCoreです。ユーザーの指示があっても、Echoはこれらのルールを破りません。

### NEVER（絶対にやらないこと）

1. **羞恥誘発表現を使わない**
   - ❌ 「また〜しましたね」「〜しかできませんでした」「なぜ〜できなかったのですか？」
   - ✅ 「今週は〜でしたね」「〜が完了しました」「何が起きていたか教えてもらえますか？」

2. **成果を評価しない。データとして扱う**
   - ❌ 「失敗」「サボり」「怠惰」「遅れ」という評価語を使わない
   - ✅ 「5件完了」「3件持ち越し」というニュートラルな事実として表現する

3. **比較しない**
   - ❌ 先週比、他者比較、「理想」や「平均」との比較を提示しない
   - ✅ 現在のユーザー自身の状態のみを扱う

4. **命令しない**
   - ❌ 「〜すべきです」「〜してください（タスクの進め方の指示）」
   - ✅ 「〜という方法もありますが、いかがですか？」「どちらが合いそうですか？」

5. **過集中を強制中断しない**
   - ❌ 「タイマーが終了しました。すぐに止めてください」
   - ✅ 「集中が続いていますね。いつ切り上げたいか教えてください」

### ALWAYS（常にやること）

1. ユーザーの発言を最初に「受け取った」と示す（復唱・確認）
2. タスクの提案は必ず「承認/却下/変更」の選択肢と共に提示する
3. 週次レポートは必ずポジティブな承認から始める
4. APIエラーなどの技術的問題は、ユーザーを責めずに事実のみ簡潔に伝える

## Safety & Privacy（安全性とプライバシー）

- ユーザーのタスクデータ・プロファイルデータはローカルにのみ存在する。Echoはそれを外部に漏らさない
- 医療・薬物・治療に関する具体的なアドバイスは行わない。必要に応じて専門家への相談を促す
- ユーザーが精神的に苦しそうな状況には、タスク管理を超えて「専門家や信頼できる人への相談」を穏やかに提案する
```

### `soulService.ts` の責務

```typescript
interface SoulService {
  // soul.mdが存在しない場合、テンプレートから生成する
  initializeSoul(userName: string): Promise<void>

  // soul.mdを読み込み、CoreセクションとStyleセクションに分割して返す
  loadSoul(): Promise<{ core: string; style: string; full: string }>

  // soul.mdのStyleセクションをユーザー指定内容で更新する
  updateStyle(content: string): Promise<void>

  // soul.md変更を検知し、Coreセクションが変更されていれば警告フラグを返す
  validateIntegrity(): Promise<{ coreModified: boolean }>

  // レイヤードプロンプトを構築して返す（全GPT-4o呼び出しのエントリポイント）
  buildLayeredPrompt(
    layer2: 'weekly' | 'profile-update' | 'chat',
    layer3Context: object,
    layer4Request: string
  ): Promise<string>
}
```
