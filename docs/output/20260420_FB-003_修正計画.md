#### FB-003-8: 週次スイープ後のClearedタスク消えない

**根本原因推定**  
sweepService.tsのcleaningフェーズ完了→IPCイベント送信→renderer側のuseTaskReducerのstate更新が非同期で噛み合っていない。

**修正方針**  
```
sweepService.ts
  └─ sweep完了時に `sweep:complete` IPCイベントを発火（clearTaskIds[] を含める）

src/preload/index.ts
  └─ `sweep:complete` イベントをexposeに追加

useTaskReducer.ts
  └─ `sweep:complete` 受信時に CLEARED タスクをstateから強制削除するアクション追加

App.tsx または Dashboard.tsx
  └─ useEffect で `sweep:complete` を購読し、reducer dispatchを呼ぶ
```

**変更ファイル**
- `src/main/services/sweepService.ts`
- `src/preload/index.ts`
- `src/renderer/hooks/useTaskReducer.ts`
- `src/renderer/App.tsx`

#### FB-003-7: タスク開始とタスク詳細定義のフラグが曖昧

**根本原因推定**  
useChat.tsのメッセージ処理がコンテキストで動的にモード判定しているため、ユーザー意図と実行結果がズレる。

**修正方針**  
```
TaskDetail.tsx
  └─ 「Echoで詳細質問」ボタン押下時に、chatSessionに mode:'detail_inquiry' フラグを設定してChatDrawerを開く

ChatDrawer.tsx
  └─ mode props を受け取り、入力プレースホルダーとヘッダーラベルを切り替える
      - detail_inquiry: "Echoに詳細を聞く..."  （ヘッダー: 🔍 詳細質問モード）
      - task_generation: "Echoに話しかける..."  （ヘッダー: 💬 タスク相談）

useChat.ts
  └─ mode が detail_inquiry の場合のみ 5W2H 質問フローを起動
  └─ それ以外は既存のタスク生成フローを維持
```

**変更ファイル**
- `src/renderer/components/TaskDetail.tsx`
- `src/renderer/components/ChatDrawer.tsx`
- `src/renderer/hooks/useChat.ts`

#### FB-003-1: TIMELINEからスイープレポートが閲覧できない

**根本原因推定**  
Timeline.tsxがIPCでレポートデータを要求した際、ローカルアーカイブのファイルI/Oが未完了、またはパス解決が不正確で、Promiseが解決されずUI上で無限ローディングになっている。

**修正方針**  
```
src/main/index.ts (または sweepService.ts)
  └─ `reports:list` IPCハンドラーを実装
      - ローカルアーカイブディレクトリを走査してレポートファイル一覧を返す
      - ファイルが存在しない場合は空配列を返す（ハングを防ぐ）

src/preload/index.ts
  └─ `reports:list` と `reports:get` をexposeに追加

Timeline.tsx
  └─ データ取得のtry-catchを強化し、エラー時は「レポートなし」状態にフォールバック
  └─ ローディング状態にタイムアウト（5秒）を設けてユーザーに通知
```

**変更ファイル**
- `src/main/index.ts`
- `src/preload/index.ts`
- `src/renderer/components/Timeline.tsx`

#### FB-003-2: ユーザーコンテキスト強化（ファイルインポート）

**設計思想（パネル議論より）**  
インポートしたファイルの内容をEchoのプロンプト構築時の`[USER CONTEXT]`セクションに組み込み、よりパーソナライズされた提案を実現する。

**修正方針**  
```
src/renderer/components/SettingsModal.tsx
  └─ 「ユーザーコンテキスト」セクションを追加
      - ファイル選択ボタン（.md / .txt のみ）
      - インポート済みファイル名の表示
      - 削除ボタン

src/main/index.ts
  └─ IPCハンドラー `context:import` を追加
      - ファイルサイズ制限: 100KB
      - 許可形式: .md, .txt
      - 読み込み内容を user_context.txt として app data に保存

src/main/services/soulService.ts
  └─ プロンプト構築時に user_context.txt が存在すれば [USER CONTEXT] セクションを追加
      - soulLayered プロンプト構造の末尾に挿入
      - 例:
        [USER CONTEXT]
        ユーザーが提供したコンテキスト情報:
        {user_context_content}

src/preload/index.ts
  └─ `context:import` と `context:get` をexposeに追加
```

**制約設計**
- ファイルサイズ上限: 100KB（大容量ファイルによるトークン過多を防ぐ）
- 対応形式: `.md`, `.txt` のみ
- 1ファイルのみ（複数ファイルは後のフェーズで対応）
- セッションごとに再読み込みせず、app dataに永続化

**変更ファイル**
- `src/renderer/components/SettingsModal.tsx`
- `src/main/index.ts`
- `src/main/services/soulService.ts`
- `src/preload/index.ts`

## 🎯 設計原則（議論から導出）

1. **明示的トリガー優先** ── AIモードの切り替えは必ずユーザーの明示アクション（ボタン押下）を起点にする。暗黙的な文脈推定はUXの不確実性を生む。
2. **経過の可視化** ── タイマーは「残り」ではなく「進捗」を見せる。達成感が次のアクションを生む。
3. **IDはアイデンティティ** ── ランダムIDは管理不能。連番IDは状態把握と運用の基盤になる。
4. **ユーザーコンテキストはEchoの記憶** ── 汎用AIではなく「私のEcho」にするための最も直接的な手段。設計は拡張性を持たせるが実装は最小スコープで。
5. **状態の整合性はシステムが保証する** ── スイープ完了後のクリーンアップは、UIのフォールバックではなくIPCイベント駆動の確実な削除で対応する。
