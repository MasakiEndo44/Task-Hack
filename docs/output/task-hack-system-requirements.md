# Task-Hack システム要件定義書

> ADHDにとって最も使いやすいタスクアプリとは何か？
> 航空管制(ATC)メタファーとAI秘書の融合による、「余白のある机上」の実現

---

## 専門家議論ログ（5ターン）

### 登場人物
- **Dr. 河村（認知科学・ADHD研究）**: ADHD当事者研究と認知負荷理論の専門家。実行機能障害のメカニズムに精通。
- **藤原エンジニア（UIアーキテクト）**: ATC（航空管制）システムのUI設計経験を持つシニアエンジニア。ミッションクリティカルUI設計のスペシャリスト。
- **みさき（ギャル・ADHD当事者）**: ADHD当事者。数々のタスク管理アプリを試しては挫折してきたリアルなユーザー代表。忖度なしのフィードバックを提供。

### ターン1：「なぜADHDはタスク管理アプリを使い続けられないのか？」

**Dr. 河村**: まず根本的な問題提起から始めましょう。ADHDの当事者がタスク管理アプリを「使い始めるが、やがて放棄する」——この現象は非常に普遍的です。原因は大きく3つの実行機能障害に起因します。

1. **ワーキングメモリの脆弱性**: 「今何をしていたか」を忘れる。アプリを開いた瞬間、何をしに来たか忘れることすらある。
2. **タスク開始困難（イニシエーション障害）**: タスクの存在は認識しているが、「始める」ための心理的エネルギーの閾値が高い。特にタスクが曖昧・巨大な場合、麻痺が起きる。
3. **時間盲（Time Blindness）**: 時間の経過を「感じられない」。30分が5分に感じられることもあれば、逆もある。締め切りという概念が感覚として理解できない。

そして最も致命的なのは、**恥の累積**です。未完了タスクが画面に赤く残り続けると、アプリを開くこと自体が罰のトリガーになる。これがいわゆる「タスク管理アプリ墓場」の正体です。

**藤原**: 航空管制の観点からすると、これは非常に興味深い。ATCシステムの設計哲学には「**情報の鮮度管理**」という概念があります。管制官のレーダースクリーンには、関係のない航空機の情報は表示されません。表示されるのは「今、自分が制御すべき機体」だけです。

管制官もまた、極度の認知負荷の下で働いています。彼らが破綻しないのは、システムが**情報の出入りを厳格に制御**しているからです。フライトストリップ——あの紙の短冊——は、1機の航空機につき1枚。物理的に手元に置ける枚数に限界がある。これが自然な「ワーキングメモリの外部化」として機能していたわけです。

**みさき**: えー、めっちゃわかるんだけど〜！ うちさ、Todoistとか最初すっごいハマるのよ。でも2週間くらいで「あ、もう無理」ってなる。なんでかっていうと、**溜まったタスクの一覧を見た瞬間に心が折れる**の。「え、これ全部やるの？ 無理じゃん」って。で、アプリ自体を開かなくなる。あとね、タスクを「書く」のがもうめんどい。書いてる間に「あ、あれもやらなきゃ」って思い出して、結局何も書き終わらないまま別のことしちゃう。

**Dr. 河村**: みさきさんの体験は教科書的なADHDの「タスク回避行動」です。重要なのは、これは意志の問題ではなく、**脳の報酬系と実行機能の構造的な問題**だということ。ここから導かれる設計原則の第一条は——

> **原則①：画面に存在するタスクの総量を、常に「安心できる量」に制限する。**

### ターン2：「ATCメタファーが解決するもの、しないもの」

**藤原**: 添付画像のアプリ「Flight Strip Todo」の設計を分析すると、ATCメタファーが非常にうまく機能している部分があります。

**機能するATCメタファー**:
- **ゾーン分割（ACTIVE / NEXT ACTION / HOLDING / CLEARED）**: 管制で言う「進入管制」「待機」「交信完了」に対応。ユーザーは今どのフェーズにいるか一目でわかる。
- **フライトストリップ（紙の短冊）型カード**: 物理的に1枚の紙に収まる情報量 = 認知負荷の自然な制限。
- **上限制限（ACTIVE max 1, NEXT max 5）**: 管制官が同時に扱えるセクター内の航空機数に上限があるのと同じ。物理的な制約が「選択の麻痺」を防ぐ。

**注意が必要な点**:
- フライトストリップの「到着→離陸→交信完了」という一方向のフローは美しいが、現実のタスクには「差し戻し」「保留」「分割」がある。ATCメタファーを厳格に適用しすぎると、現実の複雑さとの乖離が生じる。

**Dr. 河村**: そこで重要になるのが「**メタファーの境界**」の設計です。ATCの世界観は維持しつつ、ユーザーが「このメタファーに縛られている」と感じないようにする。具体的には——

- **HOLDING（待機）ゾーン**: 管制では「着陸待ちの旋回」ですが、Task-Hackでは「今日はやらないが、忘れてはいけないもの」の置き場所として再定義する。重要なのは、HOLDINGにあるタスクは**視覚的に後退している**こと。薄い色、小さいフォント、折りたたみ可能。存在は認識できるが、注意を引かない。
- **CLEARED（完了）ゾーン**: これが最も重要。一般的なTodoアプリは完了タスクにチェックマークを付けて残す。しかしADHDにとっては、完了タスクすら「ノイズ」になりうる。

**みさき**: ほんとそれ！ 完了にチェック入れた瞬間は気持ちいいのよ。でも次の日見たら「え、昨日3個しかやってないの…」ってなって逆に凹む。**完了タスクは消えてほしい、でも「やった記録」はどこかに残っていてほしい**。矛盾してるけど、これがリアル。

**藤原**: 素晴らしい指摘です。ATCでも、着陸した航空機のストリップはラックから外されます。画面から物理的に消える。でもフライトレコーダーにはすべて記録されている。これをTask-Hackに適用するなら——

> **原則②：CLEAREDタスクは画面から速やかに「離陸」させ、ナレッジベース（Obsidian）に「着陸」させる。画面は常に「余白のある机上」を維持する。**

**Dr. 河村**: ここにAI秘書の最初の仕事が生まれます。「完了タスクのアーカイブ化」を人間に任せると、それ自体がタスクになってしまう。週次でAIが自動的にスイープし、ドキュメント化して画面から消す。ユーザーが意識することなく、机の上は常にきれいに保たれる。

> **原則③：「片付け」はユーザーの仕事ではない。AIが担う。**

### ターン3：「AIチャットとタスク生成——入力のハードルをゼロに近づける」

**みさき**: ここが一番大事なとこなんだけどさ、タスクを「入力する」っていう行為がそもそもしんどいのよ。たとえば「来週のプレゼン資料を作る」——これをTodoアプリに書こうとすると、「え、何から書けばいいの？ タイトルは？ 期限は？ カテゴリは？ 優先度は？」って聞かれて、**もうその時点でやる気なくなる**。

うちが欲しいのは、「友達にLINEで愚痴るみたいに」タスクを投げられる場所。「あ〜来週プレゼンあるんだけどまじだるい」って打ったら、AIが「OK、プレゼン資料作成ね。いつまで？」って聞いてくれて、会話の中から自然にタスクができあがるの。

**Dr. 河村**: これは認知科学的に非常に重要なポイントです。従来のタスク管理アプリは「**構造化された入力**」を要求します。タイトル、詳細、期限、優先度——これらを正しいフィールドに正しい形式で入力する必要がある。しかしADHDの脳は「構造化」が最も苦手な領域です。

一方、「チャット」は**非構造化入力**です。思いついたことをそのまま言語化するだけでいい。構造化はAIの仕事。これが「入力のハードルをゼロに近づける」ということの本質です。

さらに、画像のペーストも同じロジックです。デザインのモックアップ、ホワイトボードの写真、スクリーンショット——これらを「見せるだけ」でタスクが生成される。言語化すら不要。

**藤原**: UIアーキテクトの視点で整理すると、タスクの作成パスは2つ。

1. **手動作成（Quick Add）**: タイトルだけを入力するミニマルなフォーム。他のフィールドは後から、あるいはAIが補完。
2. **AIチャット経由**: 自然言語または画像を投げ、対話を通じてタスクが生成される。確認後にボード上に「着陸」する。

ここで設計上の重要な決定があります。**チャットUIの配置場所**です。

- **案A（サイドパネル）**: 常にアクセス可能。ただし画面幅を圧迫する。
- **案B（フローティングボタン → オーバーレイ）**: 画面を圧迫しない。ただしアクセスまで1クリック余分。
- **案C（下部ドロワー）**: モバイルのメッセンジャー的操作感。親しみやすい。

**みさき**: 絶対Cがいい！ LINEっぽい感じ。上にタスクのボードがあって、下からシュッとチャットが出てくるの。で、AIと話しながらタスク追加していく感じ。**画面遷移しないのがめっちゃ大事**。別の画面に行ったら何してたか忘れるから。

**Dr. 河村**: 画面遷移の排除は、ADHDの文脈記憶（コンテキストメモリ）の弱さを考慮した正しい判断です。

> **原則④：タスクの入力は「対話」で行う。構造化はAIの責務。ユーザーは思考を言語化（または画像化）するだけでよい。**
> **原則⑤：すべての操作は1画面内で完結する。画面遷移は文脈記憶の破壊を意味する。**

### ターン4：「時間盲への処方箋——タイマーとタイムラインの設計」

**Dr. 河村**: ADHDにおける「時間盲（Time Blindness）」は、最も過小評価されている障害の一つです。これは「時計が読めない」という意味ではありません。**時間の経過を身体的に感じ取ることができない**ということです。

ポモドーロテクニックは一般的な処方箋ですが、ADHDに対しては修正が必要です。

- **標準の25分/5分は硬直的すぎる**: ADHDの集中力は日によって大きく変動する。「良い日」は60分でも集中できるし、「悪い日」は10分が限界。可変長のタイマーが必要。
- **タイマー切れの通知は「強制中断」であってはならない**: 過集中（ハイパーフォーカス）状態を唐突に切断すると、再開困難になる。「あと2分」のラップアップ通知が先に来るべき。
- **タイマーは「罰」ではなく「リズム」**: 時間切れが赤く点滅するUIは不安を煽る。代わりに、穏やかな色の変化や音で「リズム」として時間の区切りを伝える。

**藤原**: タイムラインについても重要な設計判断があります。添付画像のアプリでは横軸の時間帯タイムラインが上部にあり、タスクブロックが配置されています。

ATCのレーダースクリーンにおけるタイムラインは「**予測と現実の照合**」のためにあります。到着予定時刻（ETA）とレーダー上の実際の位置を見比べて、遅延やコンフリクトを発見する。

Task-Hackのタイムラインも同様の役割を持つべきです:

- **「今日の全体像」を一目で把握**: 何時に何をする予定で、今どこにいるのかを視覚的に表示。
- **「現在時刻」の強調**: 画面上を移動するカーソルライン（ATCの「sweep line」に相当）で、「今」の位置を常に可視化する。
- **空き時間の視覚化**: スケジュールの隙間が「余白」として見えることで、「今日はまだ余裕がある」「もういっぱいだ」が直感的にわかる。

**みさき**: タイムラインあるのめっちゃいいじゃん！ でもさ、ひとつ言いたいのは、**予定通りにいかなかった時にタイムラインが「崩れる」のがストレス**なのよ。14時にやる予定だったのに16時になっちゃった、みたいな。そうするとタイムラインがぐちゃぐちゃになって、「あーもう今日はダメだ」ってなっちゃう。

**Dr. 河村**: 非常に鋭い。これは「**計画の破綻による全日放棄（All-or-Nothing Thinking）**」というADHDに顕著な思考パターンです。対策は——

- タイムラインは「予定」ではなく「**ログ**」として設計する。つまり、「何時に何をした」の記録であり、「何時に何をすべき」の計画ではない。
- あるいは、予定と実績の両方を表示するが、ずれが発生しても**視覚的にネガティブに見えないデザイン**にする。ずれは自然なことだ、と体感させる。

> **原則⑥：時間は「管理するもの」ではなく「感じるもの」として設計する。タイマーはリズムを刻み、タイムラインは物語を描く。計画からのずれを罰しない。**

### ターン5：「AI秘書の成長——ナレッジの蓄積と『もう一人の自分』」

**みさき**: ここまでの話、全部いいんだけどさ、最後に一個だけ。うちがマジで欲しいのは、**「自分のことを覚えていてくれるAI」**なの。毎回「あなたは誰ですか、何の仕事してますか」からスタートするChatGPTじゃなくて、「あ、みさきさん、先週のプレゼン資料どうなりました？ 今週は確か●●の締め切りがありましたよね」って言ってくれるやつ。

だってさ、ADHDって**自分のことすら忘れる**のよ。先週何やったか覚えてない。先月の目標とか完全に宇宙の彼方。だから「あなたは先月こういう目標を立てていましたよ」って教えてくれる存在がいたら、めちゃくちゃ助かる。

**Dr. 河村**: みさきさんが言っているのは、認知科学で言う「**自伝的記憶の外部化**」です。ADHDの当事者は、エピソード記憶（いつ何をしたか）の想起が定型発達者より有意に困難であることが報告されています。

AI秘書がユーザープロファイル（MDファイル群）を蓄積・更新し続けることで、ユーザーの「外部自伝メモリ」として機能する——これは単なる便利機能ではなく、**ADHDの根幹的な困りごとに対するソリューション**です。

**藤原**: 技術的な設計として、これを実現するために以下の構造を提案します:

```
Obsidian Vault/Task-Hack/
  ├── weekly-reports/
  │   ├── 2026-W16_report.md    ← 週次レポート
  │   └── ...
  ├── user-profile/
  │   ├── identity.md            ← 基本情報・仕事内容・強み
  │   ├── patterns.md            ← 行動パターン（集中しやすい時間帯等）
  │   ├── goals.md               ← 短期/中期/長期目標
  │   └── insights.md            ← AIが見つけた傾向や気づき
  └── archive/
      ├── 2026-W15_tasks.md      ← スイープされたタスクのアーカイブ
      └── ...
```

週次レポート生成のフローは:

1. **Collect**: CLEAREDゾーンのタスクデータを取得
2. **Analyze**: ユーザープロファイル（identity.md, patterns.md, goals.md）を読み込み
3. **Generate**: 週次レポートMDを生成（完了タスクサマリー、達成度、発見した傾向）
4. **Update**: レポートの分析結果をもとにuser-profile内の各MDファイルを更新
5. **Sweep**: CLEAREDタスクをアーカイブMDに変換し、メイン画面のJSONから削除

**Dr. 河村**: このサイクルが回るたびに、AIはユーザーのことをより深く理解していきます。「金曜日の午後は集中力が落ちる傾向がある」「デザイン関連のタスクは過集中に入りやすい」「月初は新しいプロジェクトの立ち上げが多い」——こういった**パーソナルな生産性パターン**がナレッジとして蓄積される。

これは単なるスケジューリングではありません。**ADHDの自己理解の支援**です。

**みさき**: ちょっと泣きそうなんだけど。マジでそういうの欲しかった。自分のことわかってくれて、助けてくれるAI。しかもタスクこなすたびに賢くなってくれるって、**育成ゲーム的な楽しさもある**じゃん！ それめっちゃ続けるモチベになるわ。

**Dr. 河村**: 最後の原則を。

> **原則⑦：AI秘書はタスクの管理者ではなく、ユーザーの「外部自伝メモリ」である。タスクをこなすたびに賢くなり、ユーザーの自己理解を深める存在として設計する。**

## システム概要

### 1. システムの目的

- **解決する具体的な課題**:
  - ADHDを持つユーザーが既存のタスク管理アプリを「使い続けられない」問題（タスク回避行動、恥の累積、入力のハードル、時間盲）
  - 完了タスクの蓄積によるUI汚染と心理的負担
  - 自分の生産性パターン・行動履歴を振り返る手段の欠如

- **目標とする成果**:
  - ADHD当事者が3ヶ月以上継続して使用できるタスク管理ツールの実現
  - AIとの協働による、タスク管理の認知コスト80%削減
  - ユーザーの行動パターンの可視化と自己理解の促進

- **期待されるビジネスインパクト**:
  - 個人利用ツールとして開発。将来的に同様の課題を持つニューロダイバーシティ・コミュニティへの展開可能性。

### 2. 主な機能

**コア機能（5個）**:
1. ATC型ダッシュボード（ACTIVE/NEXT ACTION/HOLDING/CLEAREDの4ゾーン + タイムライン）
2. タスク数上限制限（ACTIVE: 1, NEXT ACTION: 5）
3. ポモドーロ/可変長タイマー（ラップアップ通知付き）
4. AIチャットによるタスク生成（自然言語 + 画像ペースト対応）
5. ローカルJSON保存とElectronデスクトップアプリ

**差別化機能（3個）**:
1. 週次AIスイープ（完了タスクの自動ドキュメント化＆画面消去）
2. AI秘書の成長機能（ユーザープロファイルMD群の継続的更新）
3. 繰り返しタスク（定期タスク）の自動生成

**付加価値機能（2個）**:
1. AIによるタスク優先度提案（現在のコンテキストとプロファイルに基づく）
2. Obsidian Vaultとのネイティブ連携（週次レポート/プロファイル/アーカイブ出力）

### 3. 提供する価値

- **ユーザーにとっての直接的価値**: 「タスク管理のストレス」そのものの消滅。画面を開いた時に安心感を感じるUI。自分を理解してくれるAIパートナー。
- **ビジネスにとっての価値**: ADHD向けツール市場への独自ポジショニング。ユーザープロファイル蓄積による高いスイッチングコスト。
- **社会的価値**: ニューロダイバーシティの生産性支援ツールとしてのモデルケース。ADHD当事者の自己理解を促進するエンパワーメントツール。

### 4. 実現方法の概要

- **主要テクノロジー**: Electron + Vite + React + TypeScript, Vanilla CSS, OpenAI API (GPT-4o)
- **重要なインテグレーション**: Obsidian Vault（ローカルファイルシステム経由）、OpenAI API（マルチモーダル対応）
- **スケーラビリティ**: 個人利用が前提。データはすべてローカル。クラウド依存はAI API呼び出しのみ。

## 技術スタック

| レイヤー | 技術 | 選定理由 |
|---------|------|----------|
| デスクトップ基盤 | **Electron** | ローカルファイルシステムへの直接アクセス（JSON, MD読み書き）が必須。Webブラウザでは不可能 |
| ビルドツール | **Vite** | 高速なHMR。Electronとの統合が成熟 (`electron-vite`) |
| UIフレームワーク | **React + TypeScript** | コンポーネントベースの設計がフライトストリップUIと相性が良い。型安全性で開発速度と品質を両立 |
| スタイリング | **Vanilla CSS (CSS Modules)** | ATCの緻密なデザイン（グロー、スキャンライン、アニメーション）をフレームワークの制約なしに実現 |
| ドラッグ＆ドロップ | **@dnd-kit/core** | React向けの軽量かつアクセシブルなD&Dライブラリ |
| AI連携 | **OpenAI API (GPT-4o)** | マルチモーダル対応（画像解析+テキスト生成）。高品質な自然言語処理 |
| データ保存 | **ローカルJSON** (`electron-store` or 直接 `fs`) | Electronのメインプロセスで管理。軽量かつバックアップが容易 |
| MD出力 | **Node.js fs モジュール** | Obsidian Vault (ローカルディレクトリ) への直接書き込み |
| 開発規律 | **Superpowers** (概念の適用) | Plan → Execute → Verify の規律を開発プロセス全体に適用 |

## リスク分析

### 技術的リスク

| リスク | 影響 | 対策 |
|--------|------|------|
| Electronのバンドルサイズが大きい | 起動速度・ディスク使用量 | Viteでのツリーシェイキング、遅延読み込みの最適化 |
| OpenAI APIの応答遅延/障害 | AI機能が一時的に使用不能 | 手動タスク管理機能はAPI非依存で完全動作。ストリーミング応答で体感速度改善 |
| D&D操作のパフォーマンス | 大量タスク時のUI描画遅延 | ゾーン上限制限がそのまま描画要素数の自然な制約となる |

### ビジネス的リスク

| リスク | 影響 | 対策 |
|--------|------|------|
| OpenAI API費用の増大 | 月額コストの予測困難 | 使用量ダッシュボードの提供。プロンプト最適化によるトークン消費削減 |

### 運用的リスク

| リスク | 影響 | 対策 |
|--------|------|------|
| ローカルデータの消失 | 全タスク・プロファイルの喪失 | Obsidian Vault連携により週次で実質バックアップ。追加でJSONの定期バックアップ機能を検討 |
| ユーザーがAPIキーを設定しない | AI機能が全く使えない | 初回起動時のオンボーディングで設定誘導。AI機能なしでもコアタスク管理は動作 |

### UXリスク

| リスク | 影響 | 対策 |
|--------|------|------|
| ATCメタファーが「カッコいいが使いにくい」と感じられる | 利用離脱 | 7原則に基づいたユーザビリティを常に優先。メタファーのために使いやすさを犠牲にしない |
| AIが不適切なタスク分割をする | ユーザーの信頼低下 | 必ず確認フローを挟む。ユーザーが編集・却下できることを明示 |
