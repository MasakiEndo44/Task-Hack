# 議論ログ：UIフレームワーク選定

## 登場人物
- 🧑💼 佐藤（フロントエンド・アーキテクト）：React歴10年。大規模SPAのコンポーネント設計が専門。
- 👨🎨 鈴木（UI/UXデザイナー）：デザインシステム構築歴8年。デザイナーとエンジニアの橋渡しが得意。
- 💅 みさき（ギャル）：実際のフロントエンド開発初学者。

## Turn 1｜現状認識・問題整理
🧑💼 佐藤: 私の経験上、Tailwindを導入した3つの大規模プロジェクトのうち2つで、`className` がカオス化して可読性が著しく落ちる問題が発生しました。特に `hover:` や `focus:` などの状態修飾子が連続すると、1行が200文字を超えることもあります。
👨🎨 鈴木: 確かに。デザインシステムをFigmaで定義しても、Tailwindのユーティリティクラスにマッピングする際、微妙なSpacingやColorのズレが生じやすいです。
💅 みさき: え待って、Tailwindってクラス名考える手間が省けるからめっちゃ便利って聞いてたんですけど〜！結局長くなったら見づらいってこと？
🧑💼 佐藤: そうですね。結局これは「スタイルのカプセル化をどこで行うか」という問題なんです。CSS Modulesはファイル単位でカプセル化しますが、TailwindはHTML要素に直接スタイルを注入するため、コンポーネントの粒度が適切でないと破綻します。

## Turn 2｜選択肢の発散
👨🎨 鈴木: では、「カプセル化の粒度」と「デザイン変更の容易さ」を評価軸にしましょう。
🧑💼 佐藤: その軸で言うと、CSS Modulesはスコープが完全に閉じているため安全ですが、グローバルなテーマ変更には弱いです。一方Tailwindは `tailwind.config.js` で一元管理するため、テーマの変更には強い。
💅 みさき: つまり、Tailwindは「全体の色とかを一気に変えやすい」けど「1個のボタンのコードが長くなる」、CSS Modulesは「コードはスッキリする」けど「全体を変えるのは面倒」ってこと？
👨🎨 鈴木: みさきさんの言う通りです。例えば、ダークモード対応を考えた場合、Tailwindなら `dark:bg-gray-800` を追加するだけで済みますが、CSS ModulesだとCSS変数（Custom Properties）を綿密に設計する必要があります。

## Turn 3｜要件・範囲の確定
💅 みさき: 私が作ってるアプリ、ボタンの種類が「Primary」「Secondary」「Danger」みたいにめっちゃあるんですけど、Tailwindで毎回長いクラス書くのマジでしんどくないですか？
🧑💼 佐藤: まさにそれが重要な実需です。Tailwind単体では辛いため、通常は `cva` (class-variance-authority) などのライブラリを組み合わせて、「バリアント（状態の組み合わせ）」として抽象化します。
👨🎨 鈴木: つまり優先度としては、①デザインシステムの忠実な再現、②開発者のタイピング負荷の軽減、となりますね。
🧑💼 佐藤: ええ。Tailwind + `cva` の組み合わせなら、CSS Modulesに近いコンポーネントレベルのカプセル化を実現しつつ、Tailwindのテーマ管理の恩恵を受けられます。

## Turn 4｜技術深掘り・リスク検討
👨🎨 鈴木: ただし、FigmaのDesign TokenとTailwindのconfigの同期をどうするかが課題です。過去の事例では、手動で同期していて結局ズレが生じました。
🧑💼 佐藤: `style-dictionary` や Figma API を使って自動生成するパイプラインを構築する必要があります。
💅 みさき: ん？つまり、デザイナーがFigmaで色変えたら、勝手にコードの方も更新されるようにするってこと？
👨🎨 鈴木: そういうことです。これを「Single Source of Truth（信頼できる唯一の情報源）」の法則と呼びます。デザインとコードのどちらか一方を正とするのではなく、大元のトークンを正として両方に分配する制約を設けるべきです。

## Turn 5｜方針統合・結論
🧑💼 佐藤: では、今回のUIフレームワーク選定における3原則をまとめましょう。
① **Component-Level Encapsulation**
   Tailwindのユーティリティは直接書かず、必ずコンポーネント（+ `cva`）で隠蔽する。
② **Single Source of Truth**
   デザインとコードのテーマ定義をFigmaのトークンで一元管理する。
③ **Zero-Config Theming**
   ダークモード等のテーマ変更はTailwindのconfigとCSS変数で自動解決する。

👨🎨 鈴木: 明日からの具体アクションです。
1. `tailwind.config.js` にFigmaのカラートークンを手動で一度マッピングする。
2. `cva` と `tailwind-merge` をインストールし、ベースとなる `Button` コンポーネントを1つ実装する。
3. 既存のCSS Modulesのボタンと、Tailwindのボタンで、コード量と保守性を比較レビューする。
💅 みさき: なんか最初は「Tailwind長くて見づらい」って思ってたけど、`cva` ってやつで包めばCSS Modulesのいいとこ取りができるってことですね！めっちゃ現実的〜！

## ✅ 議論まとめ
| 項目 | 方針 |
|---|---|
| UIフレームワーク | Tailwind CSS を採用 |
| 状態管理・バリアント | `cva` + `tailwind-merge` を併用しコンポーネントに隠蔽 |
| デザイン同期 | Figmaトークンを `tailwind.config.js` の正とする |
| 最初のステップ | ButtonコンポーネントのPoC（概念実証）実装と比較 |
